# RBK Command Center — Project Context

## Overview

Next.js 16 app serving as a unified operations dashboard for Rebecca (RBK) and Emily. Deployed to Firebase Hosting at **https://rbk-cmd-center.web.app**, with Supabase as the database and Firebase Auth (Google OAuth) for authentication.

**Tech stack:** Next.js 16 + React 19 + Tailwind CSS (v4, `@tailwindcss/typography`) + Tiptap (rich text) + Supabase + Firebase (Auth, Hosting, Cloud Functions). Fonts: Geist (UI), Source Serif 4 (home greeting). Viewers auto-redirect to `/home` on login; owners/assistants land on `/` (Dashboard).

### API Routes (`app/api/`)

- **`agenda-items/route.ts`** — CRUD for agenda items (GET with batch email/topic lookup, POST supports email/topic/manual item_type with optional title — email_id and topic_id are explicitly nullable so manual items save without an email link, PATCH with bulk reorder + tags/title fields, DELETE). DB check constraint: `item_type IN ('email', 'topic', 'manual')`. **Phase 3:** All methods require auth + workspace_id scope.
- **`agenda-items/topics/route.ts`** — Recurring topics management
- **`agenda-notes/route.ts`** — CRUD for agenda notes (GET, POST, PATCH, DELETE). POST allows standalone action notes (no `email_id` or `agenda_item_id` required when `type='action'`). PATCH supports `completed` field.
- **`emails/status/route.ts`** — Email status updates (status, action_status, draft_status, priority, reminder_date, revision_comment, tbd_suggestion, tbd_notes, draft_reply, edited_draft, is_unread). Supports explicit `is_unread` field; if not provided, defaults to `false` on any PATCH (backward compatible). Used by mark-as-read on email open.
- **`emails/[id]/flag/route.ts`** — Flag emails for meeting, update meeting_notes
- **`emails/[id]/draft/route.ts`** — Draft editing and approval
- **`emails/[id]/send/route.ts`** — Send emails via Gmail API. No approval gate — sends as long as draft has content.
- **`emails/[id]/archive/route.ts`** — Archive emails in Gmail (removes INBOX label)
- **`emails/compose/route.ts`** — Compose and send a new email (not a reply). POST with `{ to, subject, body }`. Builds HTML MIME message with RBK's signature, sends via Gmail API `users/me/messages.send`. Returns `{ success, messageId }`. Used by the Compose modal in Dashboard.
- **`emails/send-batch/route.ts`** — Batch email sending. No approval gate — sends as long as drafts have content.
- **`gmail/process-email/route.ts`** — POST with `{ emailId }`. Adds "RBK/Done" label (creates it if needed) and removes INBOX label from the Gmail message. Called fire-and-forget from Dashboard on task completion and email send. Never blocks the caller on failure.
- **`gmail/thread/[threadId]/route.ts`** — GET fetches full raw Gmail thread on demand. Uses signed-in user's access token from session cookie (same auth pattern as `emails/[id]/send`). Accepts either a Gmail thread ID or message ID as the path param — tries thread fetch first, falls back to message metadata lookup to resolve the real thread ID, then fetches that thread. Returns `{ threadId, messages: GmailThreadMessage[] }` with each message containing id, threadId, from, to, subject, date (ISO), body (full decoded MIME — prefers text/html, falls back to text/plain, handles multipart recursively via base64url decode), bodyType ('html' | 'text'), snippet. Messages sorted by date ascending. Returns JSON error on failure (401/403/404/500) — never throws. Requires `gmail.readonly` scope.
- **`gmail/sync/route.ts`** — POST triggers on-demand Gmail triage. Calls the `manualTriage` HTTP Cloud Function (`https://us-central1-rbk-cmd-center.cloudfunctions.net/manualTriage`), which publishes to the `firebase-schedule-triageGmail-us-central1` Pub/Sub topic internally. Returns `{ success: true/false }`. Used by the "Sync Gmail" button on the All Emails page.
- **`gmail/sync-drafts/route.ts`** — POST triggers on-demand draft sync. Calls the `manualSyncDrafts` HTTP Cloud Function (`https://us-central1-rbk-cmd-center.cloudfunctions.net/manualSyncDrafts`), which publishes to the `firebase-schedule-syncDraftsReady-us-central1` Pub/Sub topic internally. Returns `{ success: true/false }`.
- **`gmail/sent/route.ts`** — GET endpoint fetches last 50 sent messages from Gmail API (`labelIds=SENT`). Uses workspace refresh token pattern: gets `workspaceId` from session, queries Supabase `workspaces` table for `gmail_refresh_token`, exchanges it for an access token via Google OAuth (`https://oauth2.googleapis.com/token`), then uses that token for all Gmail API calls (always fetches workspace owner's sent mail, not the logged-in user's). Fetches each message in parallel, extracts Subject/To/Date headers and body text (recursive MIME walk, prefers text/plain, falls back to stripped HTML). Returns `{ messages: SentMessage[] }` where each has: id, threadId, subject, to, date (ISO), snippet, body.
- **`gmail/trash-email/route.ts`** — POST with `{ emailId }`. Looks up `message_id` from Supabase, calls Gmail `messages.trash` (reversible). Returns `{ success: true }` even on Gmail failure (fire-and-forget). Used when marking emails as junk.
- **`calendar/today/route.ts`** — Fetch Google Calendar events for a given date
- **`calendar/create/route.ts`** — Create calendar events
- **`calendar/delete/route.ts`** — Delete calendar events
- **`auth/session/route.ts`** — Session management. Preserves existing Google access token across periodic Firebase ID token refreshes. Cookie set to 24h. **Multi-workspace:** Fetches ALL `workspace_members` rows for the Firebase UID (not just one). Falls back to email lookup for placeholder user_ids and auto-replaces them. Fetches workspace details (name, modules, module_config) for all memberships in a single batch query. Builds a `workspaces` array in the session cookie: `[{ id, name, role, modules, module_config }]`. Default workspace prefers `owner` role over `assistant`. Session cookie shape: `{ idToken, accessToken, user, workspace_id (default), role, display_name, modules, module_config, workspaces[] }`. Falls back gracefully if workspace tables don't exist yet.
- **`auth/session/workspace/route.ts`** — Lightweight GET endpoint that reads workspace fields (`workspace_id`, `role`, `modules`, `module_config`, `display_name`, `workspaces[]`) from the httpOnly `__session` cookie and returns them as JSON. Used by `AuthProvider` to populate `WorkspaceContext` on the client. Returns the full `workspaces` array so the client can offer a workspace switcher.
- **`auth/switch-workspace/route.ts`** — POST with `{ workspaceId }`. Reads current `__session` cookie, verifies user is a member of the requested workspace (checks session `workspaces[]` first, falls back to Supabase `workspace_members` query as safety net), re-sets the `__session` cookie with the new active `workspace_id` and matching role/modules/module_config from the workspaces array. Returns `{ success: true }`. Called by `AuthProvider.switchWorkspace()` before page reload.
- **`auth/signout/route.ts`** — Sign out
- **`health/route.ts`** — Health check
- **`admin/workspace-members/route.ts`** — GET fetches all workspace members with workspace name join. Admin-only (checks email === rglassberg@saracademy.org, 403 otherwise). Returns `{ members: Array<{ id, email, display_name, role, workspace_id, workspace_name, allowed_modules }> }` sorted alphabetically.
- **`admin/workspace-members/[id]/route.ts`** — PATCH updates `allowed_modules` JSONB on a workspace_members row. Admin-only. Accepts `{ allowed_modules: Record<string, boolean> | null }`. Returns `{ success: true }`.
- **`home/week-events/route.ts`** — GET fetches SAR school calendar public iCal feed (`accalendars@saracademy.org`), parses VEVENT blocks, filters to current week (Mon–Sun), returns up to 8 events as `{ events: Array<{ id, title, startTime, isAllDay, location? }> }`. 5-min cache. Non-fatal (returns empty array on failure). No auth required.
- **`home/route.ts`** — GET endpoint powering the /home landing page for non-RBK users. Requires auth. Returns `{ firstName, projects, dashboards }`. `firstName` derived from `workspace_members.display_name` (falls back to email prefix). `projects` fetched where `assignee_email = currentEmail OR currentEmail = ANY(team_emails)`, excludes archived, each row has `isMine: boolean`. `dashboards` built from effective modules (intersection of workspace modules and member `allowed_modules` via `getEffectiveModules()`), mapped to static metadata (name, description, tone). Projects dashboard always included regardless of module flags.
- **`important-docs/route.ts`** — Important docs CRUD (GET, POST, PUT, DELETE)
- **`projects/route.ts`** — Projects CRUD (GET excludes archived, POST, PATCH multi-field update, DELETE soft-archives by setting `status='archived'`)
- **`gemara/route.ts`** — Gemara resources CRUD (GET ordered by created_at desc, POST, DELETE hard-delete by id query param). SQL for `gemara_items` table is in a comment at the top of the file.
- **`simchas/route.ts`** — GET fetches the SAR Bar/Bat Mitzvah public iCal feed (`barbatmitzvah@saracademy.org`), parses VEVENT blocks manually (no npm deps), filters to current week (Mon–Sun), returns `{ events: SimchaEvent[] }`. 5-min cache via `next.revalidate`.
- **`calendar/week/route.ts`** — GET fetches Google Calendar events for the full current week (Mon–Sun) from `kraussb@saracademy.org`. Used by the Simchas & Shivas page to find shiva/funeral events across the whole week (not just today).
- **`admissions/route.ts`** — GET fetches admission applications AND current students for re-enrollment from Veracross API. Two OAuth2 token requests: (1) admissions token with scopes `admission.applications:list admission.applications:read admission.applicants:list admission.applicants:read admission.households:list` — do NOT add `admission.households:read`. (2) students token with scopes `students:list students:read admission.applications:list admission.applicants:list admission.households:list`. Both use `VERACROSS_ADMISSIONS_CLIENT_ID`/`SECRET`. Paginates applications and students (1000/page). Applications filtered to `year_applying_for === 2026`, excludes HS grades (9-12) and unprocessed (application_status === 0). Students filtered to `grade_level` -3 to 7, `enrollment_status` in [2,3,4,5,6,8]. Returns `{ applications: AdmissionApplication[], reEnrollments: ReEnrollmentStudent[] }`. Students fetch is non-fatal (returns empty array on failure). No city data (fetched separately). Refresh button resets admissionsData + admissionsCities + increments refreshKey. Page title: "Admissions & Enrollment". UI has three tabs: Admissions Overview (summary pills [Accepted, Waitlisted, Withdrawn — Denied pill removed], family response breakdown, by-grade bars, Two-column grid (items-start): left column has Family Response card (badge + bar scaled to max count + count) + City pie chart card (legend above, 280px pie below); right column has By Grade card (bars scale to max grade count). City pie: 15-color palette, hover tooltips, top 14 + "Other" slice.), Enrollment Projection (stat cards for Enrollment Complete/Accepted Offer/Considering/Pipeline Total, horizontal funnel bar, by-grade breakdown table with badges + mini progress bars, collapsible declined families section), and New Enrollment (confirmed/pending pills). Drill-down panel (420px) with real names.
- **`admissions/cities/route.ts`** — GET accepts `?applicantIds=1,2,3` query param. Scopes: `admission.applicants:list admission.applicants:read admission.households:list` (do NOT add `:read`). Uses paginated LIST endpoints (not individual fetches) for performance: fetches ALL applicants via `/v3/admission/applicants` (paginated, 1000/page) to get household_id, then ALL households via `/v3/admission/households` (paginated) to get city. Joins applicant→household→city, filters to requested IDs. Only 2-4 API calls total instead of 400+. Household city field is `data.city`. Returns `{ cities: Record<number, string> }`. Any failure returns `{ cities: {} }`. Called from Dashboard after main admissions data loads; stored in separate `admissionsCities` state.
- **`admissions/applicants/route.ts`** — POST accepts `{ applicantIds: number[] }`, fetches each applicant from Veracross `/v3/admission/applicants/{id}` in parallel (batches of 20), returns `{ applicantNames: Record<number, string> }` with "First Last" names. Called lazily from Dashboard when the drill-down panel opens — only fetches names for the currently filtered view, caches in state to avoid re-fetching.
- **`admissions/reenrollment/route.ts`** — GET fetches re-enrollment data from Veracross `/v3/academics/enrollments`. Authenticates with `academics.enrollments:list` scope. Paginates, filters to 2026 school year and excludes HS grades. Returns `{ enrollments: ReEnrollment[] }` with person_id, first_name, last_name, grade_level, enrollment_status, school_year. **ON HOLD** — route file exists but the Re-enrollment tab UI has been removed from Dashboard.tsx pending Veracross API endpoint confirmation.
- **`absences/route.ts`** — GET fetches today's student absences from Veracross API. Authenticates via OAuth2 client credentials (`VERACROSS_CLIENT_ID`, `VERACROSS_CLIENT_SECRET`) with scopes `master_attendance:list students:list`. Paginates through all master_attendance pages AND all student pages using `X-Page-Number` header (API caps at 1000/page, school has ~1731 students). Builds a `gradeMap` (Map<person_id, grade_level_id>) from the `/v3/students` endpoint (the master_attendance endpoint does NOT include grade_level). Converts numeric grade_level IDs to display labels via `GRADE_LABELS` map (40=Infant/Toddler, 35=2 Year Nursery, 30=3 Year Nursery, 25=4 Year Nursery, 20=Kindergarten, 1-8=1st-8th Grade). Excludes HS by `grading_period !== 29`, excludes present students (`attendance_category === 0`). After fetching live data, queries Supabase `attendance_cache` for YTD absence counts (`attendance_category = 1`) and consecutive absence streaks (walks backwards through cached school days). Returns `{ date, total, absences, tardies, earlyDismissals }` with each record containing `person_id, name, status_code, status_label, excused, attendance_category, notes, late_arrival_time, early_dismissal_time, grade_level, grade_level_id, ytd_absences, consecutive_absences`. Cache queries wrapped in try/catch — defaults to 0 if cache is empty.
- **`absences/sync/route.ts`** — GET syncs Veracross attendance to Supabase `attendance_cache` table. Protected by `Authorization: Bearer ${SYNC_SECRET}` header (401 without). Two modes via `?mode=` query param: `daily` (default) fetches today's attendance, filters out HS, upserts all records (including present) with `onConflict: 'person_id,attendance_date'`; `backfill` loops from 2025-09-02 to yesterday (skipping weekends), fetches each day, upserts, 300ms delay between requests. Both modes also fetch paginated students for grade_level_id lookup.
- **`lever/route.ts`** — GET fetches all Lever recruiting data in parallel: postings (published + internal), opportunities (limit 100, `expand=applications` so each application is an object with `posting` field), and stages. Auth via `LEVER_API_KEY` (Basic auth). Filters out SAR High School postings/teams. Returns `{ postings, opportunities, stages }`.
- **`webhook/lever/route.ts`** — POST webhook for Lever events. Processes `applicationCreated`: fetches opportunity + posting, skips HS, sends Slack DM to RBK (U04NBR22Y) with applicant name, role, and Lever link. Always returns 200. **To activate:** Lever Settings > Integrations & API > Webhooks, add URL `https://rbk-cmd-center.web.app/api/webhook/lever`, enable `applicationCreated`.
- **`webhook/email/route.ts`** — Inbound email webhook from Google Apps Script. Accepts `EmailWebhookPayload` with auth via `WEBHOOK_SECRET`. On insert, extracts from name/email, parses date, stores to `emails` table (including `draft_reply` when present). **Phase 2:** Stamps `workspace_id: 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'` (RBK seed workspace) on every new email insert. Handles duplicates (23505) by updating `draft_status` AND `draft_reply` if provided — so draft content from Emily's "Drafts Ready" sync lands in the DB even when the email already exists from a prior triage sync. **Priority remapping:** When incoming `priority === 'drafts_ready'` (not a valid DB enum), remaps to `priority: 'rbk_action'` and sets `draft_status: 'draft_ready'` so the email lands in the Drafts Ready section.

## Current Features

### Email Inbox
- **Unified `ExpandedEmailPanel` component** — single shared expanded view used everywhere (EmailCard, all inbox sections, popup modal). Root div is `flex flex-col h-full`; inner two-column layout is `flex-1 min-h-0` with thread side (`flex-[4]`, `overflow-y-auto flex-1`) and draft side (`flex-[5]`, `flex-1 min-h-0`). Both sides fill available height and scroll independently. Draft textarea is **uncontrolled** (`defaultValue={draftValue}`) with a `draftTextRef` (useRef) for tracking edits — avoids Dashboard re-renders on every keystroke. `onChange` writes to `draftTextRef.current`; `onBlur` auto-saves to Supabase; "Mark Ready" reads from the ref. The `key={selectedEmailObj.id}` on ExpandedEmailPanel forces remount on email switch so `defaultValue` resets correctly. Left panel card clicks set `draftTextRef.current` to the selected email's draft content. Includes "Discard" button (red outline) alongside Send button in Drafts popup to reset draft content.
- **Thread support** — emails with matching `thread_id` are grouped and displayed chronologically in the left panel. Shows "Thread (N)" label for multi-message threads, "Original Email" for single emails. Each message shows sender name, timestamp, and cleaned body separated by dividers. Thread emails are fetched from Supabase if not all present in local state (handles threads beyond the 500-email fetch limit).
- **`linkifyText` helper** — converts URLs in email body text to clickable `<a>` tags (blue, underlined, opens in new tab). Applied in ExpandedEmailPanel thread view and task panel email body.
- **Three-zone Gmail-like layout** — Zone 1: Top bar with category `<select>` dropdown (RBK Action, Review, Important No Action, Invitations, FYI, Untagged, Emily Action, Sent), Drafts Ready button, TBD button, Sync Gmail button. Zone 2: Left panel (w-[380px]) with search input and scrollable compact email card list (priority dot, sender, subject, time ago). Zone 3: Right panel (flex-1, flex-col h-full) with sticky header (subject + sender on left, action icon row on right, border-b) above scrollable `ExpandedEmailPanel`. Action icons: Mark Done, Mark Urgent, TBD, Remind Me, Add to Agenda (star), Add to Calendar, Add Task — same handlers as EmailCard. Empty state shows envelope icon. `emailCategory` state (line 268) controls which derived array is shown; `expandedEmail` state selects the email for the right panel. Category dropdown shows live counts per category. `ExpandedEmailPanel` is rendered with `key={selectedEmailObj.id}` so it remounts on email switch (resets `defaultValue` in the draft textarea). Left panel card clicks set `draftTextRef.current` to the selected email's draft content. Separate `draftText` useState is kept for the Drafts Ready popup modal textarea (different context, no focus loss issue there).
- **Sent category** — "Sent" option in category dropdown fetches last 50 sent messages from Gmail API via `GET /api/gmail/sent` on first selection. State: `sentEmails` (array), `sentLoading` (boolean), `sentLoadedRef` (useRef, guards single fetch). Left panel cards show "To: recipient" instead of sender, no priority dot. Right panel shows a simple header (subject + "To:" + date) and full-width body text (`whitespace-pre-wrap`) — no ExpandedEmailPanel, no draft textarea, no action icons. Search filters sent emails by subject or recipient.
- **Mark as read/unread** — clicking an email card optimistically sets `is_unread: false` and fires a PATCH to `/api/emails/status`. "Mark Unread" envelope icon appears in the right panel action row when `is_unread === false`; clicking it sets `is_unread: true`.
- Priority color coding with dots from `priorityConfig` map
- Draft editing with approve/revision workflow (Request Revision sends to Emily's queue with comment)
- Send via Gmail API with batch support — no approval gate, sendable as long as draft has content
- **Gmail processing on completion/send** — when a task is completed or an email is sent, the connected email is: (1) marked 'done' in Supabase (optimistic update), (2) labeled "RBK/Done" in Gmail, (3) archived (INBOX label removed). Only fires for email-connected tasks, not standalone/agenda-note-only tasks. Gmail calls are fire-and-forget (never block the action).
- Archive functionality (marks done in Supabase + archives in Gmail)
- Snooze with reminder dates (hides email until date)
- Search across all fields (subject, from, summary, body) — filters the active category
- Bulk selection: floating action bar for selected emails (mark done, clear)
- Shiva emails (`action_status === 'shiva'`) are excluded from all inbox priority sections but are **not** shown in the inbox — they appear on the Simchas & Shivas page instead.
- Snoozed emails filtered out of main lists

### Firebase Cloud Functions (`functions/src/index.ts`)
- **`manualTriage`** — HTTP Cloud Function (`https://us-central1-rbk-cmd-center.cloudfunctions.net/manualTriage`). CORS-enabled POST endpoint that publishes to the `firebase-schedule-triageGmail-us-central1` Pub/Sub topic, triggering `triageGmail` on demand. Called by `gmail/sync/route.ts` (Sync Gmail button).
- **`manualSyncDrafts`** — HTTP Cloud Function (`https://us-central1-rbk-cmd-center.cloudfunctions.net/manualSyncDrafts`). CORS-enabled POST endpoint that publishes to the `firebase-schedule-syncDraftsReady-us-central1` Pub/Sub topic, triggering `syncDraftsReady` on demand. Called by `gmail/sync-drafts/route.ts`.
- **`triageGmail`** — Pub/Sub scheduled function, runs every 15 min weekdays (ET). **Multi-workspace:** Queries all workspaces where `gmail_refresh_token IS NOT NULL` and loops over each one. Per workspace: exchanges refresh token for access token, fetches unprocessed inbox emails via Gmail API (`labelIds=INBOX` filter on list query), skips any message with the `DRAFT` label in its `labelIds` array (prevents Emily's draft replies from being triaged as inbound mail), runs OpenAI `gpt-4o-mini` triage (priority, summary, action_needed, draft_reply), writes to Supabase `emails` table scoped to that workspace's id, applies Gmail labels. Per-workspace error handling — if one workspace fails (e.g. expired token), logs the error and continues to the next. Env vars: `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `OPENAI_API_KEY` (in `functions/.env`). `WORKSPACE_ID` env var no longer needed. 1. Running `firebase deploy --only hosting` (ignoring the expected replace error)
2. Manually finalizing the hosting version via Firebase Hosting REST API
3. Creating a new release pointing to the finalized version
4. Restoring `allUsers` IAM on the Cloud Run service (GCP org policy strips it on each deploy)
